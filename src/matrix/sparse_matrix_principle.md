# CSC与CSR稀疏矩阵存储格式原理详解

## 一、什么是稀疏矩阵

稀疏矩阵是指矩阵中绝大多数元素为0的矩阵。在实际问题中，大规模矩阵往往是稀疏的，稀疏度甚至可达90%以上。如果用普通的二维数组存储，大量0值会浪费巨大的存储空间。因此，我们需要高效的稀疏矩阵存储格式，**只存储非零元素及其位置信息**，从而大幅减少内存占用。

目前最常用的稀疏矩阵存储格式包括COO、CSR和CSC。其中，**CSR（Compressed Sparse Row，压缩稀疏行）** 和 **CSC（Compressed Sparse Column，压缩稀疏列）** 是应用最广泛的两种格式。


## 二、预备示例矩阵

为了便于理解，我们用一个具体的稀疏矩阵 `A` 作为示例：

```
    列0  列1  列2  列3
行0 [ 1   0   0   2 ]
行1 [ 0   0   3   0 ]
行2 [ 4   0   5   0 ]
行3 [ 0   6   0   7 ]
```

该矩阵共有 **8个非零元素**：1, 2, 3, 4, 5, 6, 7。


## 三、CSR（压缩稀疏行）格式

### 3.1 核心思想

CSR的核心思想是：**按行**读取矩阵，只存储非零的数据，把“空着”的地方挤掉。CSR使用三个一维数组来存储矩阵：

| 数组 | 含义 | 长度 |
|------|------|------|
| `values` | 按行存储所有非零元素的值 | nnz（非零元个数） |
| `col_indices` | 每个非零元素对应的列索引 | nnz |
| `row_ptr` | 每行在`values`中的起始偏移位置 | 行数 + 1 |

### 3.2 存储过程图解

```
原始矩阵 A（4×4）：
┌─────────────┐
│ 1  0  0  2  │  ← 行0
│ 0  0  3  0  │  ← 行1
│ 4  0  5  0  │  ← 行2
│ 0  6  0  7  │  ← 行3
└─────────────┘

按行扫描非零元素：
行0: 1(列0), 2(列3)
行1: 3(列2)
行2: 4(列0), 5(列2)
行3: 6(列1), 7(列3)

      ↓ 压缩存储 ↓

values      = [ 1,  2,  3,  4,  5,  6,  7 ]
              ↑   ↑   ↑   ↑   ↑   ↑   ↑
            行0  行0  行1  行2  行2  行3  行3

col_indices = [ 0,  3,  2,  0,  2,  1,  3 ]
              ↑   ↑   ↑   ↑   ↑   ↑   ↑
            列0  列3  列2  列0  列2  列1  列3

row_ptr     = [ 0,  2,  3,  5,  7 ]
              ↑   ↑   ↑   ↑   ↑
            行0  行1  行2  行3  结束
            起始  起始  起始  起始  (总个数)
```

### 3.3 row_ptr 详解

`row_ptr` 是最难理解的部分。它不直接存行号，而是存每一行在 `values` 中的**起始下标**：

- `row_ptr[0] = 0`：行0从 `values[0]` 开始
- `row_ptr[1] = 2`：行1从 `values[2]` 开始（行0有2个元素）
- `row_ptr[2] = 3`：行2从 `values[3]` 开始（行1有1个元素）
- `row_ptr[3] = 5`：行3从 `values[5]` 开始（行2有2个元素）
- `row_ptr[4] = 7`：总非零元素个数（行3有2个元素）

**第 i 行的非零元素**位于：
- `values[row_ptr[i] : row_ptr[i+1]]`
- `col_indices[row_ptr[i] : row_ptr[i+1]]`

例如第2行（i=2）：从 `row_ptr[2]=3` 到 `row_ptr[3]=5`，对应 `values[3]=4, values[4]=5`，列索引为 `col_indices[3]=0, col_indices[4]=2`，即 (4, 列0) 和 (5, 列2)。

### 3.4 空间复杂度

对于一个 `m × n` 的矩阵，若有 `a` 个非零元素，CSR格式需要的存储单元为 **2a + m + 1**。


## 四、CSC（压缩稀疏列）格式

### 4.1 核心思想

CSC是CSR的**列方向对应版本**——按列压缩存储。如果把矩阵转置后做CSR，就得到了CSC。CSC同样使用三个一维数组：

| 数组 | 含义 | 长度 |
|------|------|------|
| `values` | 按列存储所有非零元素的值 | nnz |
| `row_indices` | 每个非零元素对应的行索引 | nnz |
| `col_ptr` | 每列在`values`中的起始偏移位置 | 列数 + 1 |

### 4.2 存储过程图解

```
原始矩阵 A（4×4）：
┌─────────────┐
│ 1  0  0  2  │
│ 0  0  3  0  │
│ 4  0  5  0  │
│ 0  6  0  7  │
└─────────────┘

按列扫描非零元素：
列0: 1(行0), 4(行2)
列1: 6(行3)
列2: 3(行1), 5(行2)
列3: 2(行0), 7(行3)

      ↓ 压缩存储 ↓

values      = [ 1,  4,  6,  3,  5,  2,  7 ]
              ↑   ↑   ↑   ↑   ↑   ↑   ↑
            列0  列0  列1  列2  列2  列3  列3

row_indices = [ 0,  2,  3,  1,  2,  0,  3 ]
              ↑   ↑   ↑   ↑   ↑   ↑   ↑
            行0  行2  行3  行1  行2  行0  行3

col_ptr     = [ 0,  2,  3,  5,  7 ]
              ↑   ↑   ↑   ↑   ↑
            列0  列1  列2  列3  结束
            起始  起始  起始  起始  (总个数)
```

### 4.3 col_ptr 详解

`col_ptr` 与CSR中的 `row_ptr` 逻辑完全一致，只是方向从行变成了列：

- `col_ptr[0] = 0`：列0从 `values[0]` 开始
- `col_ptr[1] = 2`：列1从 `values[2]` 开始（列0有2个元素）
- `col_ptr[2] = 3`：列2从 `values[3]` 开始（列1有1个元素）
- `col_ptr[3] = 5`：列3从 `values[5]` 开始（列2有2个元素）
- `col_ptr[4] = 7`：总非零元素个数（列3有2个元素）

**第 j 列的非零元素**位于：
- `values[col_ptr[j] : col_ptr[j+1]]`
- `row_indices[col_ptr[j] : col_ptr[j+1]]`

### 4.4 空间复杂度

对于一个 `m × n` 的矩阵，若有 `a` 个非零元素，CSC格式需要的存储单元为 **2a + n + 1**。


## 五、CSR vs CSC 对比总结

| 对比维度 | CSR | CSC |
|----------|-----|-----|
| **压缩方向** | 按行压缩 | 按列压缩 |
| **三个数组** | `values`, `col_indices`, `row_ptr` | `values`, `row_indices`, `col_ptr` |
| **指针数组** | `row_ptr`（每行起始位置） | `col_ptr`（每列起始位置） |
| **索引数组** | `col_indices`（列号） | `row_indices`（行号） |
| **空间复杂度** | 2a + m + 1 | 2a + n + 1 |
| **适用场景** | 按行遍历（如矩阵-向量乘法） | 按列遍历（如矩阵转置） |
| **访问效率** | 行访问高效 | 列访问高效 |

### 5.1 如何选择？

- 如果算法主要**按行访问**矩阵（如行优先的矩阵-向量乘法），优先使用 **CSR**
- 如果算法主要**按列访问**矩阵（如列优先操作或矩阵转置），优先使用 **CSC**
- CSR和CSC可以互相转换，且转换效率较高

### 5.2 记忆口诀

> **CSR = 行压缩**（Row → `row_ptr` 指行）
> **CSC = 列压缩**（Column → `col_ptr` 指列）


## 六、与COO格式的对比

COO（Coordinate Format）是最简单的稀疏矩阵存储格式，每个非零元素用三元组 `(行号, 列号, 数值)` 表示。

| 对比 | COO | CSR/CSC |
|------|-----|---------|
| 存储方式 | 每个元素独立存储三元组 | 压缩存储，共享行/列指针 |
| 空间效率 | 较低（3a个单元） | 较高（约2a + m/n个单元） |
| 随机访问 | 任意元素可直接定位 | 需通过指针数组间接访问 |
| 适用场景 | 构建矩阵、格式转换 | 高效计算 |

**COO的优缺点**：
- ✅ 优点：格式简单，利于稀疏格式之间的快速转换
- ❌ 缺点：不能直接进行算术运算


## 七、总结

CSR和CSC是稀疏矩阵存储中最重要的两种压缩格式。它们通过**三个一维数组**巧妙地实现了对稀疏矩阵的高效存储：

1. **只存非零值**，大幅节省内存
2. **用指针数组**（`row_ptr` / `col_ptr`）记录每行/每列的起始位置
3. **用索引数组**（`col_indices` / `row_indices`）记录每个值的位置

选择CSR还是CSC，取决于你的算法是**行优先**还是**列优先**——选对了方向，就能在内存和计算效率上都获得最优表现。
